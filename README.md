# Концепция программного продукта «Photo Studio Booking»

## 1. Описание проблемы

*Проблема*
Учёт бронирований залов фотостудии ведётся вручную, из‑за чего возникают накладки по времени, потеря данных о клиентах и сложность подсчёта загрузки залов и выручки за период.

*Воздействует на* 
администратора студии, владельца студии, клиентов (фотографов и заказчиков съёмок).

*Результатом чего является* 
двойные брони на один зал, простои залов, ошибки в цене (зал + оборудование), отсутствие оперативной статистики по залам.

*Выигрыш от ПП* 
единый локальный каталог залов и оборудования; автоматический расчёт цены и проверка занятости слотов (09:00–21:00); список всех броней с редактированием и отменой; отчёт по загрузке и выручке за выбранный период; быстрый старт с демо-данными (`studio.db`).


## 2. Назначение программного средства

*Какие системы смогут применять ПС для решения своих задач?*
Небольшие фотостудии и арендные павильоны с несколькими залами, где один сотрудник ведёт расписание с рабочего ПК.

*От каких систем/пользователей поступают входные данные ПС?* 
От администратора (пароль, данные клиента, выбор зала/даты/времени/оборудования); справочная структура данных задаётся схемой БД.

*Кто/что использует результаты (Потребители, включая другие системы и сервисы)?*
Администратор (интерфейс ПС); клиент получает услугу вне ПС (звонок/визит); локальная БД хранит результаты операций.

*Какие сторонние системы/сервисы участвуют в процессе обработки данных?* 
В базовой версии внешние API и облачные сервисы не требуются. Участвуют: ОС (файловая система), SQLite/JDBC, runtime JavaFX.

## 4. Анализ конкурентов

Создаваемый программный продукт не является уникальным по своей рыночной нише, поэтому перед нами стоит задача выявления и реализации особых функций и конкурентных преимуществ. Для этого проведём анализ прототипов, ориентированных на использование на персональных компьютерах либо широко применяемых администраторами фотостудий в повседневной практике. В процессе формирования таблицы сравнения были использованы следующие условные обозначения для сегмента рынка:

- U – небольшие студии с одним-двумя залами и минимальным штатом администраторов;
- P – профессиональные студии с тремя и более залами, расширенным перечнем оборудования и услуг;
- XS – сети фотостудий, объединяющие несколько территориально распределённых локаций.

Проведённый анализ показал, что представленные на рынке решения либо перегружены функциональностью и дороги для небольших студий (1С), либо ориентированы исключительно на облачное взаимодействие и не обеспечивают надёжной работы в офлайн-режиме (YClients), либо не предоставляют средств автоматического контроля конфликтов бронирования и учёта оборудования (Excel). Разрабатываемая информационная система бронирования фотостудий направлена на заполнение выявленной ниши, предлагая автономное десктопное приложение, ключевыми преимуществами которого станут автоматизированная проверка доступности залов и оборудования, исключение накладок в расписании, гибкий учёт оснащения и формирование отчётности по загрузке.

## 5. Описание будущего программного продукта

**Photo Studio Booking** – это настольное приложение, созданное для администраторов небольших фотостудий, которые ведут расписание залов с одного компьютера и ежедневно сталкиваются с ручным учётом броней и риском пересечения по времени. Приложение объединяет учёт залов, оборудования и бронирований в одном инструменте: оно автоматически проверяет доступность временных слотов, рассчитывает итоговую стоимость с учётом аренды зала и выбранного оборудования, а также формирует наглядный отчёт по загрузке. В отличие от электронных таблиц и общих календарей, не привязанных к конкретным залам и тарифам студии, наш продукт хранит данные локально в SQLite, не требует сервера и сразу поставляется с демо-каталогом залов — это позволяет быстро освоить систему и оценить её возможности без долгой настройки.

## 4. Сценарии использования (use case)

#### UC-01 Войти в систему

- **Название:** Войти в систему
- **Актор:** Администратор
- **Предусловие:** Установлены JDK 17, приложение запущено; БД доступна.
- **Основной поток:**
  1. Система показывает экран аутентификации.
  2. Администратор вводит пароль.
  3. Проверка пароля в таблице `admin`.
  4. При успехе, открывается Главное меню.
- **Альтернативные потоки:** 3a. Неверный пароль — сообщение «Неверный пароль», остаёмся на входе.
- **Постусловие:** Администратор авторизован, открыто главное меню.

#### UC-02 Создать бронь

- **Название:** Создать бронь
- **Актор:** Администратор
- **Предусловие:** Администратор вошёл; в БД есть хотя бы один зал.
- **Основной поток:**
  1. На вкладке «Создать бронь» выбрать зал в таблице.
  2. Нажать «Выбрать и забронировать» .
  3. Система отображает окно бронирования.
  4. Ввести имя и телефон клиента.
  5. Выбрать дату (ограничения: не прошлое, ≤30 дней).
  6. Просмотреть расписание слотов; выбрать начало и окончание из доступных.
  7. Отметить оборудование в списке.
  8. Система показывает расчёт цены.
  9. Подтвердить.
- **Альтернативные потоки:** 9a. Нарушение правил (прошлая дата, пересечение, <30 мин до начала) — сообщение пользователю.
- **Постусловие:** В БД активная бронь и связи с оборудованием.

#### UC-03 Отменить бронь

- **Название:** Отменить бронь
- **Актор:** Администратор
- **Предусловие:** Существует бронь со статусом `active`.
- **Основной поток:**
1. Перейти на вкладку «Бронирования», выбрать строку с бронью.  
2. Нажать кнопку «Отменить бронь».  
3. Подтвердить действие в диалоге.  
4. Система устанавливает статус `cancelled`.  
4. Таблица бронирований обновляется.
- **Альтернативные потоки:** 1a. Бронь уже `cancelled` — предупреждение.
- **Постусловие:** Бронь отменена, слоты освобождены для новых броней.

#### UC-04 Сформировать отчёт по загрузке

- **Название:** Сформировать отчёт по загрузке
- **Актор:** Администратор (руководитель)
- **Предусловие:** Вход выполнен.
- **Основной поток:**
	1. Перейти на вкладку «Отчёт по загрузке».  
	2. Указать даты «от» и «до».  
	3. Нажать «Сформировать».  
	4. Система выполняет агрегацию.  
	4. Система отображает таблицу: зал, количество броней, общее число часов, выручка.
- **Альтернативные потоки:** 2a. Дата «от» позже «до» — предупреждение.
- **Постусловие:** Отчёт отображён (без экспорта в файл — не реализовано).

# Систематизация функциональных требований

## 1. Матрица классификации требований

Для приложения **Photo Studio Booking** выделены следующие роли:  
- **Администратор** (он же Руководитель – объединены в одном входе) – выполняет все операции.  
- **Клиент** – косвенный пользователь, данные о нём вносит администратор, с системой не взаимодействует.  
- **Система** – включает локальную БД SQLite, логику проверки слотов, транзакции, фоновые валидации.

## 3. Приоритеты по шкале MoSCoW
- **Must (M)** – FR-01, FR-02, FR-03, FR-04, FR-05, FR-06, FR-07, FR-09, FR-10, FR-11, FR-13, FR-14, FR-15.  
  *Без этих требований система неработоспособна или теряет основную ценность (бронирование, справочники, аутентификация, базовая валидация).*
- **Should (S)** – FR-08 (редактирование брони), FR-12 (отчёты).

- **Won't (W)** – отсутствуют (всё заявленное входит в MVP).
