# Спецификация требований к программному обеспечению «Photo Studio Booking»

---

## 1. Введение

### 1.1 Цель

Целью данной спецификации является определение требований к программному обеспечению **Photo Studio Booking** — системе учёта и бронирования залов фотостудии. Документ фиксирует функциональные и нефункциональные требования, ограничения и интерфейсы.

### 1.2 Область применения

Программное обеспечение разрабатывается для использования администратором фотостудии на персональном компьютере. Приложение работает автономно: данные хранятся в локальном файле `studio.db`. Клиенты студии не входят в систему; администратор создаёт бронь, указывая имя и телефон клиента.

### **1.3 Определения, акронимы и сокращения**

- **ПС** — программное средство (приложение Photo Studio Booking).
- **Зал (hall)** — съёмочное помещение с атрибутами: название, вместимость, базовая цена за час, описание интерьера.
- **Бронь (booking)** — запись о резервировании зала на дату и интервал времени для клиента; статус `active` или `cancelled`.
- **Оборудование (equipment)** — дополнительная аренда (вспышки, софтбоксы и т.д.) с ценой за час; привязывается к брони.
- **DAO** — слой доступа к данным (JDBC).
- **JDK** (Java Development Kit) — комплект разработчика на языке Java.
- **JavaFX** — платформа графического интерфейса.
- **MVP** – Minimum Viable Product (минимально жизнеспособный продукт).
- **ФР** — функциональное требование.
- **НФР** — нефункциональное требование.
- **БД** – база данных.
- **SQLite** — локальная реляционная база данных на устройстве.
- **MoSCoW** — метод приоритизации требований (Must, Should, Could, Won’t).
- **CRUD** — Create, Read, Update, Delete — четыре базовые операции управления данными (создание, чтение, обновление, удаление)

## **2. Общее описание**

### **2.1 Продукт**

Программное обеспечение должно представлять собой настольное приложение для администратора фотостудии: учёт залов и оборудования, создание и ведение бронирований, просмотр расписания занятости, отчёт по загрузке и выручке за период. Данные хранятся локально на компьютере пользователя; работа без постоянного подключения к интернету. При запуске пользователь проходит вход в систему, после чего получает доступ к основным разделам через единое главное окно с навигацией по задачам (бронирование, справочники, отчёты).

### **2.2 Функции**

Программное обеспечение должно иметь следующие функциональные возможности:

- аутентификация администратора по паролю;
- просмотр каталога залов и создание брони для клиента;
- просмотр расписания зала на дату (занятые/свободные часовые слоты 09:00–21:00);
- просмотр всех бронирований, редактирование активных, отмена;
- CRUD залов;
- CRUD оборудования;
- отчёт по загрузке залов за период: число броней, часы, выручка;
- выход из системы с возвратом на экран входа.

### **2.3 Целевая аудитория**

Программное обеспечение предназначено для использования администратором (сотрудником студии, ведущим расписание). Конечные клиенты студии взаимодействуют с системой только опосредованно — через данные, вводимые администратором.

## **3. Требования**

### **3.1 Функциональные требования**

Приоритеты указаны по методу MoSCoW: **M** — Must have, **S** — Should have.

1. *ФР-01* (M): Система должна аутентифицировать администратора по паролю.
2. *ФР-02* (M): Система должна отображать список залов с вместимостью, ценой и описанием.
3. *ФР-03* (M): Пользователь может создать бронь с именем, телефоном, датой, временем и оборудованием.
4. *ФР-04* (M): Система должна рассчитывать итоговую стоимость брони.
5. *ФР-05* (M): Система должна сохранять бронь зала и оборудования в одной транзакции (целостность).
6. *ФР-06* (M): Система должна предлагать только непересекающиеся слоты в рабочих часах (09:00–21:00).
7. *ФР-07* (M): Пользователь может просмотреть все бронирования с деталями зала и оборудования.
8. *ФР-08* (S): Пользователь может редактировать бронь только со статусом `active`.
9. *ФР-09* (M): Пользователь может отменить только активную бронь.
10. *ФР-10* (M): Пользователь может добавить, изменить и удалить зал (CRUD).
11. *ФР-11* (M): Пользователь может добавить, изменить и удалить оборудование (CRUD).
12. *ФР-12* (S): Пользователь может получить отчёт за период (загрузка, выручка).
13. *ФР-13* (M): Пользователь может выйти из системы.
14. *ФР-14* (M): Система должна отклонять бронь при дате в прошлом или более чем на 30 дней вперёд.
15. *ФР-15* (M): Система должна отклонять бронь длительностью вне диапазона 1–8 часов или вне временного окна 09:00–21:00.

**Бизнес-правила:**

- дата брони: не в прошлом, не более чем на 30 дней вперёд;
- для брони на сегодня: начало не ранее чем через 30 минут от текущего момента;
- длительность: от 1 до 8 часов;
- рабочие часы студии: 09:00–21:00.

### **3.2 Нефункциональные требования**

1. *НФР-01* (Производительность): Время отображения списка бронирований и справочника залов не должно превышать 2 секунд при условии, что в локальной базе данных содержится не более 5 000 записей о бронированиях.
2. *НФР-02* (Производительность): Построение отчёта по загрузке за период до 12 месяцев должно выполняться не дольше 3 секунд при максимальном количестве залов — 20 и бронирований — 5 000.
3. *НФР-03* (Производительность): Проверка доступности временных слотов при создании нового бронирования должна занимать не более 1 секунды.
4. *НФР-04* (Надёжность): При возникновении ошибки доступа к локальной базе данных на этапе запуска приложение должно отобразить понятное пользователю сообщение о проблеме и корректно завершить работу, гарантируя отсутствие повреждения файла БД.
5. *НФР-05* (Надёжность): Операция сохранения бронирования вместе с выбранным оборудованием должна быть атомарной: либо все данные успешно сохраняются, либо изменения полностью отменяются без частичной фиксации.
6. *НФР-06* (Удобство): Создание бронирования должно занимать не более 5 целенаправленных действий пользователя, начиная с главного экрана: выбор зала → открытие диалога создания → подтверждение (промежуточные шаги, такие как выбор даты/времени и оборудования, не увеличивают общий счёт действий сверх установленного лимита).
7. *НФР-07* (Переносимость): Приложение должно работать на операционных системах Windows 10/11 и Linux при наличии установленной JDK/JRE версии 17 или выше без необходимости перекомпиляции исходного кода.
8. *НФР-08* (Сопровождаемость): Архитектура приложения должна обеспечивать возможность добавления нового типа отчёта или нового поля в справочники без изменения существующей логики бронирования.
9. *НФР-09* (Хранение): Система обязана хранить полную историю бронирований (включая отменённые) в локальной базе данных в течение срока не менее **12 месяцев** с момента создания бронирования, за исключением случаев ручного удаления данных пользователем.

## **4. Интерфейсы**

### **4.1 Интерфейс пользователя**

Интерфейс должен быть понятным администратору без специальной подготовки. Реализуется как оконное приложение JavaFX (программная вёрстка, единое главное окно с вкладками). 
Основные разделы: 
- «Вход» (пароль), 
- «Главная» (краткая сводка),
- «Создать бронь» (каталог залов и форма бронирования), 
- «Бронирования» (таблица, редактирование активных, отмена), 
- «Залы» и «Оборудование» (справочники CRUD), 
- «Отчёт» (период, таблица: зал, число броней, часы, выручка). 
Таблицы — с сортировкой по столбцам где уместно; ошибки валидации — понятные сообщения в диалогах.

### **4.2 Интерфейс аппаратуры**

Программное обеспечение должно корректно взаимодействовать с локальной файловой системой. Специальных периферийных устройств не требуется.

### **4.3 Интерфейс программного обеспечения**

**Внешние API и сетевые сервисы** не являются обязательной частью базовой версии продукта.

## **5. Требования к производительности**

### **5.1 Временные требования**

Отклик на действия пользователя на локальном ПК: 
- открытие списков и справочников — до 2 с; 
- отчёт за период — до 3 с; 
- проверка слотов при бронировании — до 1 с. 
- Сообщение о критической ошибке БД при запуске — не позднее 5 с после старта приложения.
### **5.2 Нагрузочные требования**

Программное обеспечение должно удерживать пиковую нагрузку одного пользователя (администратор). Многопользовательский режим не входит в базовый объем работ.
Устойчивая работа при: 
- до 20 залов; 
- до 50 позиций оборудования; 
- до 5 000 броней в истории (активные и отменённые); 
- до 30 одновременных броней на один зал в один день (в пределах слотов 09:00–21:00). 
При превышении порогов допустимо плавное снижение скорости без потери данных и без сбоев записи.

## **6. Требования к поддержке**

### **6.1 Управление изменениями**

В случае необходимости внести изменения, процесс изменения программного обеспечения должен включать обновление `schema.sql`, соответствующих DAO/моделей и UI; автоматизированные миграции БД не требуются в базовой версии.

### **6.2 Требования к тестированию**

Программное обеспечение должно проходить все необходимые тесты перед выпуском в промышленную эксплуатацию:
- Модульное тестирование – покрытие ключевой бизнес-логики:
	- сервис бронирования – проверка расчёта стоимости, валидации дат и времени, правильности определения доступных слотов;
	- сервис аутентификации – проверка пароля;
	- DAO-слоя (с использованием тестовой БД SQLite в памяти).
- Интеграционные/ручные сценарии — вход, CRUD залов и оборудования, создание брони с оборудованием, конфликт слотов, редактирование/отмена, отчёт за неделю, выход.
- Приёмочное тестирование – демонстрация заказчику (преподавателю) всех реализованных функций согласно спецификации.

## **7. Аналоги и зависимости**

*Аналоги*: электронные таблицы (гибко, но без автоматической проверки конфликтов); YClients (онлайн-запись, слабый офлайн); 1С (мощно, дорого для малых студий). Photo Studio Booking ориентирован на небольшую студию: локальный офлайн, учёт залов и оборудования, проверка пересечений слотов, отчёт по загрузке без сервера и подписки.

Программное обеспечение должно быть совместимо с:
- **JDK 17+**;
- **Apache Maven 3.x** (сборка);
- **SQLite** (через драйвер `org.xerial:sqlite-jdbc`).

## **8. Ограничения**

### **8.1 Лицензирование**

Программное обеспечение должно использоваться в соответствии с лицензионным соглашением используемых библиотек (OpenJFX, SQLite JDBC, SLF4J — открытые лицензии). Отдельная лицензия на сам продукт в репозитории не указана.

**Ограничения базовой версии:**

- один администратор;
- в минимальной версии допускается локальная аутентификация;
- роль «клиент» и онлайн-бронирование не входят в первую поставку;
- механизм резервного копирования БД может выполняться внешними средствами.

## **9. Отказы**

### **9.1 Устойчивость к отказам**

Программное обеспечение должно быть устойчиво к сбою подключения к БД при старте (блокировка запуска с сообщением). Ошибки валидации бизнес-правил обрабатываются исключениями с показом пользователю. Ошибки удаления зала/оборудования при связях — с сообщением в UI.

### **9.2 Восстановление после отказов**

В случае отказа программного обеспечения, необходимо предусмотреть механизмы восстановления работы: повторный запуск приложения; при повреждении `studio.db` — восстановление из резервной копии файла (вне приложения).

## **11. Приложения**

### **11.1 Инструкции по установке**

Инструкции по установке программного обеспечения должны быть предоставлены в формате **README**:

1. Установить JDK 17 и Maven.
2. Клонировать/распаковать проект.
3. Выполнить `mvn clean javafx:run` или `mvn clean package` и `java -jar target/studio-booking-1.0.0.jar`.
4. Войти с паролем `admin` (после первого запуска).

### **11.2 Примеры использования**

Примеры использования программного обеспечения должны быть представлены в виде **сценариев** (см. `task1_2.md`): вход администратора, создание брони с оборудованием, отмена брони, отчёт за неделю.
